## 16. Issues Discovered & Resolved During Validation

The following 8 issues were identified during validation testing, root-caused, patched, and re-verified before final sign-off. All 284 tests in the final run reflect the post-fix state.

---

### ESC-RACE-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | Critical |
| **Description**     | Concurrent dispute and expiry on the same escrow within the same second. Both state transitions attempted simultaneously, resulting in an inconsistent escrow state. |
| **Root Cause**      | No row-level lock on the escrow record during state transition. Two concurrent requests could both read the current state, pass validation, and attempt conflicting writes. |
| **Fix Applied**     | Added SELECT FOR UPDATE on the escrow row before any state change, serializing all concurrent transitions on the same escrow record. |
| **Verification**    | Verified with 50 concurrent requests targeting the same escrow in a load test. All requests correctly serialized; no inconsistent states observed across 500 iterations. |

---

### ESC-RACE-002

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | High |
| **Description**     | Release and dispute arriving in the same event loop tick. The release completes first, then the dispute finds the escrow in an unexpected state, producing an unhandled error rather than a clean rejection. |
| **Root Cause**      | State transition logic did not re-check the current state after acquiring the lock. The dispute handler assumed the state it read before the lock was still valid. |
| **Fix Applied**     | Added a guard clause that re-checks the current escrow state immediately before transition. If the state has changed since the initial read, the handler returns HTTP 409 Conflict with a descriptive error. |
| **Verification**    | Replayed the exact interleaving scenario in integration tests. The dispute request now receives a clean 409 Conflict response. Confirmed in 200 rapid-fire dual-request test runs. |

---

### AUTH-WINDOW-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | Medium |
| **Description**     | Daily spend aggregation was off by one second at midnight UTC rollover. Transactions processed at exactly 00:00:00.000 UTC could be attributed to either the previous or current day depending on query timing. |
| **Root Cause**      | The time window boundary used second-level precision (`>= '2026-04-14 00:00:00'`), leaving a one-second ambiguity zone at the boundary. |
| **Fix Applied**     | Switched to microsecond precision on time window boundaries with an explicit inclusive-start, exclusive-end range (`>= '2026-04-14T00:00:00.000000' AND < '2026-04-15T00:00:00.000000'`). |
| **Verification**    | Inserted transactions at midnight boundary (00:00:00.000000, 23:59:59.999999, 00:00:00.000001) and confirmed correct day attribution in all cases. |

---

### AUTH-CONCURRENT-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | Critical |
| **Description**     | Two payment requests for the same agent within 10ms both read daily spend as $900 (limit $1,000), both authorized $200. Total $1,100 exceeds the configured daily limit. |
| **Root Cause**      | The spend calculation read the current aggregate without any locking. Two concurrent reads could both see the pre-update total and both authorize, causing the limit to be exceeded. |
| **Fix Applied**     | Introduced an advisory lock keyed on agent_id + org_id during spend calculation. Only one spend check can execute at a time for a given agent within an organization. |
| **Verification**    | Load test with 20 concurrent $200 authorization requests against a $1,000 daily limit. Exactly 5 authorized, 15 declined. Total spend never exceeded $1,000 across 100 test iterations. |

---

### WEBHOOK-RETRY-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | Medium |
| **Description**     | When a webhook endpoint was re-enabled after being disabled due to consecutive failures, the failure counter was not reset. The next single failure immediately re-disabled the endpoint, making recovery effectively impossible without manual database intervention. |
| **Root Cause**      | The re-enable operation set `is_active = true` but did not reset the `consecutive_failures` counter. The counter remained at the threshold value (e.g., 5), so one more failure pushed it over the limit again. |
| **Fix Applied**     | The re-enable operation now resets `consecutive_failures` to 0 alongside setting `is_active = true`, giving the endpoint a clean slate. |
| **Verification**    | Disabled an endpoint via 5 consecutive failures, re-enabled it, then triggered one more failure. Confirmed the endpoint remained active with `consecutive_failures = 1`. |

---

### WALLET-PRECISION-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | High |
| **Description**     | Floating point arithmetic causing $0.01 discrepancies over 1,000+ transactions. Cumulative rounding errors in balance calculations caused wallet balances to drift from the expected values. |
| **Root Cause**      | Balance operations were using standard floating point arithmetic, which introduces binary representation errors on decimal currency values (e.g., 0.1 + 0.2 != 0.3). |
| **Fix Applied**     | All balance operations now use fixed-precision decimal arithmetic with explicit rounding to 2 decimal places at each operation boundary. Stored values use integer-cent representation internally. |
| **Verification**    | Ran a synthetic test of 10,000 sequential transactions with varied amounts. Final balance matched expected value to the cent. Repeated with randomized transaction ordering -- zero discrepancies. |

---

### REC-TIMEZONE-001

| Field               | Detail |
| ------------------- | ------ |
| **Severity**        | Medium |
| **Description**     | Reconciliation window was calculated using the server timezone instead of the organization's configured timezone. Organizations in UTC+4 or UTC-8 had their reconciliation window misaligned, causing transactions near the boundary to appear in the wrong day's report. |
| **Root Cause**      | The reconciliation engine hard-coded UTC for window calculation rather than reading the organization's `timezone` configuration field. |
| **Fix Applied**     | Reconciliation window is now calculated from the organization's configured timezone. The start and end timestamps are converted to UTC for the actual query, preserving the organization's logical day boundary. |
| **Verification**    | Tested with organizations configured for UTC+4 (Dubai), UTC-8 (Los Angeles), and UTC+5:30 (Mumbai). All reconciliation reports correctly reflected transactions within each organization's local day. |

### RLS-RECURSION-001

| Field               | Detail |

| **Severity**        | Critical |
| **Description**     | RLS policies referencing a view that itself had RLS policies, causing infinite recursion under concurrent admin queries. The database would return a stack depth limit error, blocking all queries on the affected tables. |
| **Root Cause**      | A row-level security policy on the organizations table referenced a helper view for permission checks.
